# 동네핀 요구사항 명세서

작성일: 2026-06-09

이 문서는 `docs/project/동네핀.pdf`를 기준으로 동네핀 MVP 백로그를 정리한다. 기존 명칭과 기획 흐름은 더 이상 사용하지 않고, 사용자가 저장한 **장소**와 길 위에 남긴 **쪽지**를 묶어 만드는 **코스**를 핵심 개념으로 둔다.

- 기준 자료: `docs/project/동네핀.pdf`
- 로컬 문서: `docs/project/dongnepin-requirements.md`
- MVP 지역 전략: 서울 동 단위 후보 지역(`망원·합정·연남`, `성수·서울숲`, `을지로·충무로`) 우선
- 하단 탭: 홈 / 지도 / + / 보관함 / 마이

이 문서는 애자일 프로젝트 백로그로 바로 옮길 수 있도록 사용자 관점의 유저 스토리와 인수 조건 중심으로 정리한다. 우선순위는 `Must`, `Should`, `Could`로 나눈다.

## 핵심 용어

| 용어 | 정의 |
|---|---|
| 장소 | TourAPI, 운영자 큐레이션, 사용자 저장 등을 통해 지도에 표시되는 실제 방문 지점이다. 관광지, 음식점, 카페, 행사, 산책 지점 등을 포함한다. |
| 쪽지 | 사용자가 길거리, 골목, 이동 중, 장소 주변 등 특정 좌표에 남기는 위치 기반 기록이다. 장소 자체는 아니며 제목, 내용, 사진, 좌표, 카테고리, 공개 범위를 가진다. |
| 코스 | 사용자가 저장한 장소와 쪽지를 순서 있게 묶은 지도 기반 경로다. 코스에는 장소와 쪽지가 모두 들어갈 수 있고, 지도 경로와 공개 범위를 가진다. |
| 보관함 | 사용자가 저장한 장소, 저장한 쪽지, 내 코스, 다녀온 기록, 친구가 공유한 항목을 관리하는 공간이다. |
| 친구 | 사용자가 관계를 맺은 다른 사용자다. 친구 공개 쪽지와 친구 공개 코스를 볼 수 있다. |
| 공개 범위 | 쪽지와 코스의 노출 범위다. `전체공개`, `친구공개`, `나만보기`를 지원한다. |

## 구현 상태 표기 규칙

| 상태 | 의미 |
|---|---|
| 구현됨 | 현재 backend repo에서 유저 스토리를 직접 만족하는 API, 도메인, 저장소, 테스트 근거가 있다. |
| 부분 구현 | 기존 Enjoy-Trip 기능으로 일부 기반은 있으나 동네핀 스토리를 그대로 만족하지는 못한다. |
| 미구현 | 현재 backend repo에서 해당 기능을 찾지 못했다. |
| 결정 필요 | MVP 범위, 데이터 정책, 외부 연동 방식 등 제품/기술 결정을 먼저 내려야 한다. |
| 문서화 | 구현 기능이 아니라 요구사항, Ready/Done 기준, 동기화 규칙을 정리하는 항목이다. |

## Epic 1. 홈에서 오늘 주변 정보 보기

| ID | 유저 스토리 | Priority | 상태 | 구현 근거 |

### US-01. 홈 상단에서 현재 계절과 날씨 보기

**As a** 오늘 주변에서 갈 곳을 찾는 사용자  
**I want** 홈 맨 위에서 현재 계절과 날씨를 바로 확인하고 싶다  
**So that** 오늘 야외 활동과 주변 탐색에 적합한지 빠르게 판단할 수 있다.

**Acceptance Criteria**

- Given 사용자가 홈에 진입하면, Then 화면 최상단에 현재 계절과 현재 날씨가 표시된다.
- Then 날씨 정보에는 최소한 날씨 상태, 기온, 위치 기준 시각이 포함된다.
- Given 날씨 API가 실패했을 때, Then 마지막으로 확인된 날씨 또는 기본 안내 문구가 표시된다.
- Then 계절과 날씨는 아래의 주변 장소 추천과 최근 쪽지 섹션보다 먼저 보여야 한다.

### US-02. 현재 위치 기준 가까운 장소를 인기순으로 보기

**As a** 지금 내 주변에서 갈 만한 곳을 찾는 사용자  
**I want** 현재 위치와 가까운 장소를 인기순으로 보고 싶다  
**So that** 멀리 검색하지 않아도 주변의 검증된 장소를 빠르게 고를 수 있다.

**Acceptance Criteria**

- Given 사용자가 위치 권한을 허용하고 홈에 진입하면, Then 현재 위치 기준 가까운 장소 목록이 표시된다.
- Then 장소 목록은 거리 기준 후보 중 인기순으로 정렬된다.
- Then 각 장소 항목에는 장소명, 대표 이미지 또는 기본 이미지, 거리, 인기 지표, 카테고리가 표시된다.
- Given 현재 위치를 사용할 수 없을 때, Then 사용자가 선택한 기본 지역 또는 서비스 기본 지역 기준 장소 목록을 표시한다.
- Given 근처 장소가 없을 때, Then 빈 상태와 지도에서 장소 찾기 CTA가 표시된다.

### US-03. 근처 최근 쪽지 보기

**As a** 주변 사람들이 방금 남긴 이야기를 보고 싶은 사용자  
**I want** 홈 하단에서 내 근처의 최근 쪽지를 보고 싶다  
**So that** 장소 검색만으로 알 수 없는 길 위의 분위기와 팁을 발견할 수 있다.

**Acceptance Criteria**

- Given 사용자가 홈에 진입하면, Then 가까운 장소 목록 아래에 근처 최근 쪽지 섹션이 표시된다.
- Then 최근 쪽지는 현재 위치 기준 접근 가능한 쪽지 중 최신순으로 정렬된다.
- Then 쪽지 항목에는 제목, 내용 일부, 사진 여부, 거리, 작성 시각, 공개 범위에 따른 작성자 표시가 포함된다.
- Then `나만보기` 쪽지는 작성자 본인에게만 표시되고, `친구공개` 쪽지는 작성자의 친구에게만 표시된다.
- Given 근처 최근 쪽지가 없을 때, Then 빈 상태와 쪽지 남기기 CTA가 표시된다.

**Backend implementation notes**

- 홈 backend는 단일 `/api/home`이 아니라 독립 API를 조합한다.
- 날씨는 기존 `GET /api/weather/briefings`를 사용하며, 외부 날씨 저장소가 실패하거나 비어도 기본 지역 fallback을 반환한다.
- 홈 상단 자연어 브리핑은 `GET /api/neighborhood/briefing?regionName=서울`을 사용한다. Spring AI/GMS 호출 실패나 저장된 공개 READY 코스 후보 부재 시에도 성공 envelope와 fallback 문장을 반환한다.
- 주변 인기 장소는 `GET /api/attractions/popular-nearby`를 사용한다. 좌표가 없으면 서울 시청 좌표와 500m 반경을 사용한다.
- 주변 최근 쪽지는 `GET /api/notes/nearby`를 사용한다. 장소와 같은 서울 시청 좌표 및 500m 기본 반경을 사용한다.
- 쪽지 생성/수정/삭제는 `POST /api/notes`, `PUT /api/notes/{id}`, `DELETE /api/notes/{id}`로 제공하며 JWT 인증 사용자의 본인 쪽지만 변경할 수 있다.

### US-04. 홈 항목에서 지도 이동하기

**As a** 홈에서 마음에 드는 장소나 쪽지를 발견한 사용자  
**I want** 홈 항목을 누르면 지도에서 바로 확인하고 싶다  
**So that** 저장하거나 코스에 추가하기 전에 위치와 주변 맥락을 볼 수 있다.

**Acceptance Criteria**

- Given 사용자가 홈의 가까운 장소 항목을 선택하면, Then 지도 화면으로 이동하고 해당 장소 좌표 중심으로 표시된다.
- Given 사용자가 홈의 최근 쪽지 항목을 선택하면, Then 지도 화면으로 이동하고 해당 쪽지 좌표 중심으로 표시된다.
- Then 바텀시트에는 선택한 장소 또는 쪽지 요약, 저장 액션, 코스에 추가 액션이 표시된다.

## Epic 2. 지도에서 장소, 쪽지, 친구 기록 탐색하기

| ID | 유저 스토리 | Priority | 상태 | 구현 근거 |
|---|---|---|---|---|
| US-05 | 지도에서 장소 핀 탐색하기 | Must | 부분 구현 | `/api/attractions`는 지역, 키워드, 좌표, 반경 검색 기반은 제공하지만 동네핀 지도 핀/바텀시트 계약은 없다. |
| US-06 | 지도에서 쪽지 핀 탐색하기 | Must | 미구현 | 쪽지 도메인, 쪽지 핀, 지도 토글 API가 없다. |
| US-07 | 지도 필터 사용하기 | Must | 미구현 | `전체 / 장소 / 쪽지 / 친구 / 저장됨` 필터와 쪽지 하위 필터 API가 없다. |
| US-08 | 마커 클러스터 보기 | Should | 미구현 | 장소/쪽지/친구 기록을 클러스터링해 집계하는 지도 응답 모델이 없다. |
| US-09 | 근처 쪽지 뽑기 | Must | 미구현 | 현재 위치 반경 50m 내 쪽지 랜덤/추천 조회 API가 없다. |

### US-05. 지도에서 장소 핀 탐색하기

**As a** 특정 동네를 둘러보고 싶은 사용자  
**I want** 지도에서 장소 핀을 보고 싶다  
**So that** 동네 안에 어떤 장소와 경험이 있는지 직관적으로 탐색할 수 있다.

**Acceptance Criteria**

- Given 사용자가 지도 탭에 진입하면, Then 현재 위치 또는 선택 지역 기준 지도가 표시된다.
- When 사용자가 지역을 선택하면, Then 해당 지역의 장소 핀이 지도에 표시된다.
- When 사용자가 장소 핀을 선택하면, Then 바텀시트에서 장소 요약, 저장 액션, 코스에 추가 액션을 확인할 수 있다.

### US-06. 지도에서 쪽지 핀 탐색하기

**As a** 다른 사람들이 남긴 로컬 이야기를 보고 싶은 사용자  
**I want** 지도에서 쪽지 핀을 보고 싶다  
**So that** 장소 리뷰가 아닌 길 위의 감각과 팁을 발견할 수 있다.

**Acceptance Criteria**

- Given 지도 화면에서 쪽지 필터가 활성화되면, Then 접근 권한이 있는 쪽지 핀이 표시된다.
- Then 쪽지 핀은 장소 핀과 구분되는 마커로 표시된다.
- When 사용자가 쪽지 핀을 선택하면, Then 제목, 내용 일부, 사진 여부, 카테고리, 작성자 공개 정보, 저장 액션, 코스에 추가 액션이 표시된다.

### US-07. 지도 필터 사용하기

**As a** 지도 정보가 많다고 느끼는 사용자  
**I want** 지도 필터로 보고 싶은 정보만 선택하고 싶다  
**So that** 장소, 쪽지, 친구 기록, 저장한 항목을 목적에 맞게 탐색할 수 있다.

**Acceptance Criteria**

- Given 지도 화면에 진입하면, Then `전체 / 장소 / 쪽지 / 친구 / 저장됨` 필터가 표시된다.
- When 사용자가 `쪽지` 필터를 선택하면, Then 쪽지 하위 필터 `전체 / 가장 좋았던 것 / 음악 / 책 / 영화 / 꿀팁 / 이동 팁`이 표시된다.
- When 사용자가 `친구` 필터를 선택하면, Then 친구가 공개한 장소/쪽지/코스 관련 핀만 표시된다.
- When 사용자가 `저장됨` 필터를 선택하면, Then 내가 저장한 장소와 쪽지만 표시된다.

### US-08. 마커 클러스터 보기

**As a** 핀이 많은 지역을 보는 사용자  
**I want** 겹친 핀을 묶어서 보고 싶다  
**So that** 지도 화면이 복잡해지지 않게 탐색할 수 있다.

**Acceptance Criteria**

- Given 줌 아웃 상태에서 장소, 쪽지, 친구 기록 핀이 겹치면, Then 클러스터 마커가 표시된다.
- Then 클러스터에는 `쪽지 12개`, `장소 8개`, `친구 기록 3개`처럼 타입별 개수가 표시된다.
- When 사용자가 줌 인하면, Then 개별 장소 핀, 쪽지 핀, 친구 프로필 마커로 분리된다.

### US-09. 근처 쪽지 뽑기

**As a** 지금 내 주변의 이야기를 알고 싶은 사용자  
**I want** 현재 위치 근처의 쪽지를 하나 뽑아보고 싶다  
**So that** 지도 전체를 탐색하지 않아도 주변의 감각적인 정보를 발견할 수 있다.

**Acceptance Criteria**

- Given 위치 권한이 허용된 상태에서, When 사용자가 근처 쪽지 뽑기를 누르면, Then 반경 50m 내 접근 가능한 쪽지 중 하나가 표시된다.
- Given 반경 50m 내 접근 가능한 쪽지가 없을 때, Then 쪽지가 없다는 빈 상태와 다른 탐색 액션이 표시된다.
- Then 표시된 쪽지는 저장하거나 새 쪽지 작성 흐름으로 이어질 수 있다.

### US-10. 쪽지 작성하기

**As a** 길 위에서 괜찮은 순간을 발견한 사용자  
**I want** 특정 좌표에 쪽지를 남기고 싶다  
**So that** 장소가 아니어도 그 순간의 감각과 팁을 기록하고 공유할 수 있다.

**Acceptance Criteria**

- Given 사용자가 하단 `+` 버튼을 누르면, Then 쪽지 작성 화면으로 이동한다.
- Then 쪽지 작성 필드는 제목, 내용, 사진 첨부, 좌표, 공개 범위를 포함한다.
- When 사용자가 필수값을 입력하고 등록하면, Then 쪽지가 저장되고 지도에 표시될 수 있다.
- Then 쪽지는 장소 엔티티로 저장하지 않고 독립적인 위치 기반 기록으로 저장한다.

### US-11. 쪽지 위치 선택하기

**As a** 정확한 위치에 기록을 남기고 싶은 사용자  
**I want** 지도에서 쪽지를 남길 좌표를 선택하고 싶다  
**So that** 특정 장소명이 없어도 길거리나 골목의 위치를 표현할 수 있다.

**Acceptance Criteria**

- Given 쪽지 작성 화면에 진입하면, Then 현재 위치 또는 기본 지도 중심 좌표가 제안된다.
- When 사용자가 지도를 움직이면, Then 지도 중심점이 쪽지 좌표로 선택된다.
- Then 선택된 좌표는 위도, 경도와 선택 지역 정보로 저장된다.

### US-12. 쪽지 공개 범위 설정하기

**As a** 내 기록의 노출 범위를 관리하고 싶은 사용자  
**I want** 쪽지마다 공개 범위를 설정하고 싶다  
**So that** 모두에게 보여줄 기록과 친구에게만 보여줄 기록, 나만 볼 기록을 구분할 수 있다.

**Acceptance Criteria**

- Given 쪽지 작성 또는 수정 화면에 있을 때, Then 공개 범위 선택 UI가 표시된다.
- Then 공개 범위는 `전체공개`, `친구공개`, `나만보기` 중 하나여야 한다.
- When `전체공개`로 저장하면, Then 모든 사용자가 해당 쪽지를 지도에서 볼 수 있다.
- When `친구공개`로 저장하면, Then 작성자의 친구만 해당 쪽지를 볼 수 있다.
- When `나만보기`로 저장하면, Then 작성자 본인만 해당 쪽지를 볼 수 있다.

### US-13. 쪽지 자동 분류하기

**As a** 쪽지를 탐색하는 사용자  
**I want** 쪽지가 관심사별로 분류되어 있기를 원한다  
**So that** 내가 원하는 감각의 정보만 빠르게 찾을 수 있다.

**Acceptance Criteria**

- Given 사용자가 쪽지를 작성하면, Then AI가 쪽지 제목과 내용을 분석해 카테고리를 제안한다.
- Then 카테고리 후보는 `가장 좋았던 것`, `음악`, `책`, `영화`, `꿀팁`, `이동 팁`, `미분류`를 포함한다.
- Then 분류 실패 시 기본 카테고리 또는 미분류 상태로 저장된다.

### US-14. 쪽지 수정/삭제하기

**As a** 내가 남긴 쪽지를 관리하고 싶은 사용자  
**I want** 쪽지 내용을 수정하거나 삭제하고 싶다  
**So that** 잘못 남긴 내용이나 더 이상 공개하고 싶지 않은 기록을 관리할 수 있다.

**Acceptance Criteria**

- Given 작성자 본인이 쪽지 상세를 볼 때, Then 수정과 삭제 액션이 표시된다.
- When 작성자가 제목, 내용, 사진, 좌표, 공개 범위를 수정하고 저장하면, Then 변경된 쪽지가 조회된다.
- When 작성자가 삭제하면, Then 지도, 보관함, 코스 추가 후보에서 더 이상 표시되지 않는다.

## Epic 4. 보관함에서 장소와 쪽지 저장하기

### US-15. 장소 저장하기

**As a** 나중에 다시 보고 싶은 장소를 발견한 사용자

**So that** 보관함에서 다시 확인하고 코스에 넣을 수 있다.

**Acceptance Criteria**

- Given 사용자가 장소 바텀시트 또는 장소 상세를 보고 있을 때, Then 저장 액션이 표시된다.
- When 사용자가 저장을 누르면, Then 장소가 내 보관함의 저장한 장소에 추가된다.
- Then 이미 저장한 장소는 저장된 상태로 표시된다.

### US-16. 쪽지 저장하기

**As a** 나중에 다시 보고 싶은 쪽지를 발견한 사용자  
**I want** 쪽지를 저장하고 싶다  
**So that** 여행 후에도 마음에 든 로컬 정보를 다시 확인하고 코스에 넣을 수 있다.

**Acceptance Criteria**

- Given 사용자가 접근 가능한 쪽지 카드를 보고 있을 때, Then 저장 액션이 표시된다.
- When 사용자가 저장을 누르면, Then 쪽지가 내 보관함의 저장한 쪽지에 추가된다.
- Then 이미 저장한 쪽지는 저장된 상태로 표시된다.
- Then 쪽지의 공개 범위가 변경되어 접근 권한이 사라지면, 보관함에서도 내용을 볼 수 없어야 한다.

### US-17. 보관함 보기

**As a** 저장한 로컬 정보를 관리하는 사용자  
**I want** 저장한 장소, 저장한 쪽지, 내 코스, 다녀온 기록, 친구가 공유한 것을 한 곳에서 보고 싶다  
**So that** 발견한 정보를 코스로 엮고 다시 방문할 수 있다.

**Acceptance Criteria**

- Given 사용자가 보관함 탭에 진입하면, Then 저장한 장소, 저장한 쪽지, 내 코스, 다녀온 기록, 친구가 공유한 것이 구분되어 표시된다.
- Given 아직 저장한 장소나 쪽지가 없을 때, Then 빈 상태와 지도 보러가기 CTA가 표시된다.
- When 사용자가 저장한 장소나 쪽지를 선택하면, Then 상세 또는 지도 바텀시트로 이동한다.

### US-18. 새 코스 만들기

**As a** 저장한 로컬 정보를 엮고 싶은 사용자  
**I want** 보관함에서 새 코스를 만들고 싶다

**Acceptance Criteria**

- Given 사용자가 보관함의 내 코스에서 새 코스 만들기를 누르면, Then 코스 생성 화면으로 이동한다.
- Then 사용자는 코스명, 지역, 공개 범위를 입력할 수 있다.
- Then 사용자는 저장한 장소와 저장한 쪽지를 선택해 코스 항목으로 추가할 수 있다.

### US-19. 코스에 장소 추가하기

- When 사용자가 코스에 추가를 누르면, Then 새 코스 생성 또는 기존 코스 선택 흐름이 제공된다.
- Then 추가된 장소는 코스 항목 타입 `장소`로 저장되고 순서를 가진다.

### US-20. 코스에 쪽지 추가하기

**As a** 감각적인 기록을 코스에 넣고 싶은 사용자  
**I want** 쪽지를 코스에 추가하고 싶다  
**So that** 장소 사이의 골목, 이동 팁, 분위기 기록까지 코스에 포함할 수 있다.

**Acceptance Criteria**

- Given 사용자가 접근 가능한 쪽지를 보고 있을 때, Then 코스에 추가 액션이 표시된다.
- When 사용자가 코스에 추가를 누르면, Then 새 코스 생성 또는 기존 코스 선택 흐름이 제공된다.
- Then 추가된 쪽지는 코스 항목 타입 `쪽지`로 저장되고 순서를 가진다.
- Then 쪽지는 장소가 아니므로 장소 ID 없이 쪽지 ID와 좌표를 기준으로 코스에 포함된다.

### US-21. 코스 순서 편집하기

**As a** 추천받거나 저장한 코스를 조정하고 싶은 사용자  
**I want** 코스의 장소와 쪽지 순서를 바꾸고 싶다

**Acceptance Criteria**

- Given 코스 상세에서 사용자가 수정하기를 누르면, Then 편집 모드로 진입한다.
- Then 사용자는 장소와 쪽지를 추가, 삭제, 순서 변경할 수 있다.
- Then MVP에서는 위/아래 이동 버튼 또는 간단한 순서 변경 방식으로 항목 순서를 바꿀 수 있다.
- When 저장하면, Then 변경된 코스 상세와 지도 경로가 표시된다.

### US-22. AI 동선 최적화하기

**As a** 여러 장소와 쪽지를 코스에 담은 사용자  
**I want** AI 동선 최적화를 받고 싶다  
**So that** 이동이 자연스러운 순서로 코스를 정리할 수 있다.

- Given 코스에 둘 이상의 위치 항목이 있을 때, Then AI 동선 최적화 버튼이 표시된다.

- Then 사용자는 제안된 순서를 적용하거나 취소할 수 있다.

### US-23. 코스 공개 범위 설정하기

### US-24. 친구 추가하기

**As a** 지인과 로컬 기록을 공유하고 싶은 사용자  
**I want** 친구를 추가하고 싶다  
**So that** 친구 공개 쪽지와 코스를 서로 볼 수 있다.

**Acceptance Criteria**

- Given 사용자가 친구 추가 화면에 진입하면, Then 닉네임 또는 식별자로 사용자를 검색할 수 있다.
- When 사용자가 친구 요청을 보내면, Then 상대방에게 요청 상태가 생성된다.
- When 상대방이 요청을 수락하면, Then 두 사용자는 친구 관계가 된다.

### US-25. 친구 목록 관리하기

**As a** 친구 관계를 관리하는 사용자  
**I want** 친구 목록과 요청 목록을 관리하고 싶다  
**So that** 공유 범위를 신뢰할 수 있는 사람에게 제한할 수 있다.

**Acceptance Criteria**

- Given 사용자가 마이페이지의 친구 관리에 진입하면, Then 친구 목록, 받은 요청, 보낸 요청이 표시된다.
- When 사용자가 친구를 삭제하면, Then 양쪽의 친구 공개 콘텐츠 접근 권한이 사라진다.
- Then 차단 또는 신고 정책은 MVP 이후 확장 항목으로 별도 결정할 수 있다.

### US-26. 친구 공개 쪽지 보기

**As a** 친구의 로컬 기록이 궁금한 사용자  
**I want** 친구가 공개한 쪽지를 지도에서 보고 싶다  
**So that** 지인의 감각과 팁을 내 탐색에 활용할 수 있다.

**Acceptance Criteria**

- Given 사용자가 지도에서 친구 필터를 선택하면, Then 친구가 `친구공개` 또는 `전체공개`로 남긴 쪽지가 표시된다.
- Then 친구가 `나만보기`로 남긴 쪽지는 표시되지 않는다.
- When 친구 관계가 끊기면, Then 기존에 보이던 친구공개 쪽지는 더 이상 조회되지 않는다.

### US-27. 친구 공개 코스 보기

**As a** 친구의 코스를 참고하고 싶은 사용자  
**I want** 친구가 공개한 코스를 보고 싶다  
**So that** 좋은 동네 코스를 저장하거나 따라갈 수 있다.

**Acceptance Criteria**

- Given 사용자가 보관함 또는 지도에서 친구가 공유한 항목을 볼 때, Then 친구 공개 코스가 표시된다.
- Then 친구 공개 코스에는 코스명, 지역, 장소 목록, 쪽지 목록, 지도 경로, 공개 범위가 표시된다.
- When 사용자가 친구 코스를 저장하면, Then 내 보관함의 친구가 공유한 것 또는 저장한 코스 후보로 확인할 수 있다.

### US-28. 공개 범위 기본값 설정하기

**As a** 반복해서 공개 범위를 고르기 귀찮은 사용자  
**I want** 쪽지와 코스의 기본 공개 범위를 설정하고 싶다  
**So that** 새 쪽지와 새 코스를 만들 때 원하는 공개 범위가 기본 선택되도록 할 수 있다.

**Acceptance Criteria**

- Given 사용자가 마이페이지에 진입하면, Then 공개 범위 기본값 설정 메뉴가 표시된다.
- Then 쪽지 기본 공개 범위와 코스 기본 공개 범위를 각각 설정할 수 있다.
- When 기본값을 변경하면, Then 이후 작성하는 쪽지와 코스 생성 화면에 해당 값이 기본 선택된다.

## Epic 7. 코스 따라가고 기록하기

### US-29. 코스 시작하기

**As a** 코스를 실제로 따라가려는 사용자  
**I want** 코스에서 시작하기를 누르고 싶다  
**So that** 홈 또는 지도 화면이 코스 진행 모드로 바뀌고 다음 행동을 확인할 수 있다.

**Acceptance Criteria**

- Given 예정된 코스 상세에서, When 사용자가 코스 시작을 누르면, Then 코스 상태가 진행 중으로 변경된다.
- Then 진행 중인 코스, 현재 위치, 다음 장소 또는 쪽지가 표시된다.
- Then 코스 지도 경로에서 현재 진행 위치를 확인할 수 있다.

### US-30. 길 위 콘텐츠 보기

**As a** 코스를 따라 이동 중인 사용자  
**I want** 목적지 사이의 이야기와 근처 쪽지를 보고 싶다  
**So that** 이동 시간이 단순한 경로가 아니라 경험의 일부가 된다.

**Acceptance Criteria**

- Given 사용자가 코스 진행 중일 때, Then 현재 위치 기준 길 위 콘텐츠가 표시된다.
- Then 반경 100m 내 접근 가능한 쪽지 또는 주변 동네 이야기를 우선 노출한다.
- Then 다음 장소까지의 이동 중 5분 경유지를 제안할 수 있다.

### US-31. 코스 항목 완료 체크하기

**As a** 코스를 따라 이동 중인 사용자  
**I want** 완료한 장소나 쪽지를 체크하고 싶다  
**So that** 다음 항목으로 자연스럽게 이동할 수 있다.

**Acceptance Criteria**

- Given 사용자가 코스 상세 또는 진행 중 홈을 보고 있을 때, Then 코스 항목 완료 액션이 표시된다.
- When 완료를 누르면, Then 해당 장소 또는 쪽지 항목이 완료 상태로 변경된다.
- Then 다음 코스 항목 안내가 표시된다.

### US-32. 다녀온 기록 남기기

**As a** 코스를 마친 사용자  
**I want** 사진과 한 줄 감상을 남기고 싶다  
**So that** 다녀온 코스를 감각적인 기록으로 보관할 수 있다.

**Acceptance Criteria**

- Given 진행 중인 코스가 있을 때, When 사용자가 코스 완료하기를 누르면, Then 코스 상태가 다녀옴으로 변경된다.
- Then 대표 사진 추가와 한 줄 감상 입력이 가능하다.
- When 사용자가 저장하면, Then 다녀온 기록 카드에 사진과 감상이 표시된다.

**I want** 내 취향을 설정하고 싶다

**Acceptance Criteria**

- Given 사용자가 취향 설정 화면에 진입하면, Then 관심 카테고리, 이동 방식, 동행 유형, 선호 분위기를 선택할 수 있다.
- When 사용자가 취향을 수정하면, Then 이후 추천 조건에 반영된다.
- Then 추천 시스템은 저장한 장소, 저장한 쪽지, 완료 코스도 취향 데이터로 활용할 수 있다.

### US-35. 알림 설정하기

**I want** 필요한 알림만 켜고 끄고 싶다  
**So that** 친구 요청과 공유 소식을 불편하지 않게 받을 수 있다.

**Acceptance Criteria**

- Given 사용자가 마이페이지 알림 설정에 진입하면, Then 친구 요청, 코스 공유, 쪽지 반응 알림 설정이 표시된다.
- When 사용자가 알림 설정을 변경하면, Then 이후 알림 발송 정책에 반영된다.

### US-36. 활동 통계 보기

**As a** 서비스를 계속 사용하는 사용자  
**I want** 내 활동 통계를 보고 싶다  
**So that** 내가 어떤 동네와 기록을 많이 남겼는지 알 수 있다.

**Acceptance Criteria**

- Given 사용자가 마이페이지에 진입하면, Then 저장한 장소 수, 작성한 쪽지 수, 만든 코스 수, 다녀온 코스 수가 표시된다.
- Then 주요 관심 카테고리와 최근 활동을 볼 수 있다.

### TS-01. 공식 관광 데이터 연동하기

**So that** 초기에도 안정적인 장소/경험 후보 데이터를 제공할 수 있다.

**Acceptance Criteria**

- Given 관광 데이터 API 키가 설정되어 있을 때, Then 지정한 지역의 관광 데이터를 수집할 수 있다.
- Then 수집 데이터에는 위치, 카테고리, 이미지, 상세 정보가 포함되어야 한다.
- Then API 실패 시 재시도 또는 실패 로그가 남아야 한다.

### TS-02. 날씨 기반 추천 정보 반영하기

**As a** 사용자  
**I want** 현재 날씨가 추천과 브리핑에 반영되기를 원한다  
**So that** 오늘 실제로 가기 좋은 곳을 판단할 수 있다.

**Acceptance Criteria**

- Given 날씨 API가 정상 응답할 때, Then 홈 브리핑, 지도 추천, 코스 추천에 날씨 정보가 반영된다.
- Given 날씨 API가 실패할 때, Then 기본 문구 또는 마지막으로 확인된 날씨 기반 fallback이 제공된다.

### TS-03. 초기 큐레이션 데이터 운영하기

**As a** 서비스 운영자  
**I want** 초기 MVP 지역의 장소, 경험 카드, 브리핑, 추천 코스 초안을 직접 관리하고 싶다  
**So that** 사용자 콘텐츠가 부족해도 서비스 품질을 유지할 수 있다.

**Acceptance Criteria**

- Given 운영자가 큐레이션 데이터를 등록하면, Then 홈, 지도, 보관함 추천, 코스 추천에 노출될 수 있다.
- Then 운영자 데이터는 AI 생성 데이터와 구분해 추적 가능해야 한다.
- Then 잘못된 추천 문구는 운영자가 수정할 수 있어야 한다.

### TS-04. 쪽지와 사용자 생성 데이터 관리하기

**As a** 서비스 운영자  
**I want** 사용자가 남긴 쪽지와 사진을 관리하고 싶다  
**So that** 부적절한 콘텐츠를 제어하고 추천 품질을 유지할 수 있다.

**Acceptance Criteria**

- Given 사용자가 쪽지를 등록하면, Then 작성자, 제목, 내용, 사진, 좌표, 카테고리, 공개 범위, 생성 시각이 저장된다.
- Then 신고 또는 숨김 처리할 수 있는 상태값을 가진다.
- Then 개인정보와 위치 데이터 보관 정책을 적용할 수 있어야 한다.

### TS-05. 공개 범위 권한 검증하기

**As a** 개발자  
**I want** 쪽지와 코스 조회 시 공개 범위를 항상 검증하고 싶다  
**So that** 친구공개와 나만보기 콘텐츠가 잘못 노출되지 않게 할 수 있다.

**Acceptance Criteria**

- Given 쪽지 또는 코스가 `나만보기`일 때, Then 작성자 외 사용자는 목록/상세/지도/검색 응답에서 볼 수 없다.
- Given 쪽지 또는 코스가 `친구공개`일 때, Then 작성자 본인과 친구만 볼 수 있다.
- Given 쪽지 또는 코스가 `전체공개`일 때, Then 로그인 사용자 또는 정책상 허용된 사용자가 볼 수 있다.
- Then 모든 목록 API와 상세 API는 동일한 공개 범위 정책을 적용한다.

### TS-06. 임베딩 대상 지역 제한하기

**As a** 개발자  
**I want** 첫 attraction embedding backfill 대상 지역을 명확히 제한하고 싶다  
**So that** 예상하지 못한 대량 데이터 처리와 잘못된 지역 처리를 방지할 수 있다.

**Acceptance Criteria**

- Given `backend/batch/src/main/resources/embedding-target-regions.yml`이 있을 때, Then 지정된 지역 외 entry가 있으면 live batch는 실행되지 않아야 한다.
- Then 동네핀 MVP 후보 지역은 서울 동 단위 후보 지역을 우선 검토한다.
- Then 기존 강릉/전주 대상 문서는 PoC 성격인지 실제 MVP 축인지 별도 결정해야 한다.

### TS-07. 이미지 업로드 관리하기

**As a** 사용자  
**I want** 쪽지에 사진을 첨부하고 코스 공유 이미지를 저장하고 싶다  
**So that** 텍스트만으로 부족한 현장 분위기를 남길 수 있다.

**Acceptance Criteria**

- Given 사용자가 쪽지를 작성할 때, Then 사진을 0개 이상 첨부할 수 있다.
- Then 첨부 사진은 쪽지 작성자, 쪽지 ID, 업로드 시각과 연결되어 저장된다.
- Given 사용자가 코스 상세에서 공유용 이미지 저장을 누르면, Then 코스명, 지역, 장소 목록, 쪽지 목록, 지도 경로를 포함한 공유 이미지 생성 또는 저장 흐름이 제공된다.

## Definition of Ready

- 유저 스토리에 사용자, 목표, 가치가 명확히 적혀 있다.
- 인수 조건이 Given/When/Then 또는 검증 가능한 문장으로 작성되어 있다.
- 필요한 데이터 출처와 API 의존성이 식별되어 있다.
- 화면 또는 API가 필요한 스토리는 관련 진입 경로가 적혀 있다.
- Must 스토리는 MVP 데모 시나리오와 연결되어 있다.
- 공개 범위가 필요한 스토리는 `전체공개`, `친구공개`, `나만보기` 중 어떤 정책을 적용하는지 명시되어 있다.

## Definition of Done

- 스토리의 모든 인수 조건을 만족한다.
- 백엔드 repo에서 구현하는 스토리는 화면 검증 대신 실제 실행 중인 백엔드에 HTTP 요청을 보내 응답 JSON을 확인한다.
- API 응답은 `success`, `data`, `error` 공통 envelope와 핵심 data field가 기대한 값과 shape로 내려오는지 검증한다.
- 생성, 수정, 삭제 기능은 요청 후 조회 API를 다시 호출해 실제 데이터 변경이 응답 JSON에 반영되는지 확인한다.
- 쪽지와 코스 조회 API는 공개 범위별 권한 검증 테스트를 포함한다.
- 빈 상태, 로딩 상태, 실패 상태가 처리되어 있다.
- 백엔드 API 또는 데이터 연동이 필요한 경우 기본 테스트가 통과한다.
- 데모 플로우에서 사용자가 다음 행동을 막힘 없이 이어갈 수 있다.

## 결정 필요 사항

- 동네핀 MVP 지역을 서울 동 단위 후보 지역으로 확정할지, 기존 강릉/전주 PoC 데이터 축을 별도 실험으로 남길지 결정해야 한다.
- 친구 기능의 MVP 범위를 `친구 요청/수락/삭제`까지만 둘지, 차단/신고까지 포함할지 결정해야 한다.
- 쪽지 사진 첨부 개수, 파일 크기, 보관 기간, 삭제 정책을 결정해야 한다.
- 쪽지와 코스의 `전체공개`가 비로그인 사용자에게도 노출되는지 결정해야 한다.
- 친구공개 콘텐츠를 코스에 추가한 뒤 친구 관계가 끊기거나 공개 범위가 바뀌었을 때 코스에서 어떻게 처리할지 결정해야 한다.
- AI 동선 최적화가 실제 경로 API를 호출할지, 좌표 기반 정렬/휴리스틱으로 MVP를 시작할지 결정해야 한다.
